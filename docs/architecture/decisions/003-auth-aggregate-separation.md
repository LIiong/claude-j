# ADR-003: Auth 聚合与 User 聚合分离策略

**日期**：2026-04-16
**状态**：已接受
**决策者**：@architect

## 背景

在实现用户认证系统时，面临聚合边界划分的决策：

1. 用户基本信息（姓名、头像、个人资料）与认证信息（密码、登录状态、锁定记录）具有不同的变更频率和事务边界
2. 认证相关的业务规则（登录失败锁定、Token 刷新）相对独立，不应影响用户基本信息的修改
3. 需要支持独立的水平扩展策略（认证服务可能面临更高并发）

## 决策

**将认证功能划分为独立的 auth 聚合，与 user 聚合通过 UserId 值对象弱关联。**

聚合划分：

| 聚合 | 职责 | 核心实体 |
|------|------|----------|
| user | 用户基本信息管理 | User（聚合根） |
| auth | 用户认证凭证与状态管理 | AuthUser（聚合根）、UserSession、LoginLog |

关联方式：
- auth 聚合中的 AuthUser 持有 `UserId` 值对象，标识所属用户
- 不建立对象引用，避免跨聚合直接操作
- user 聚合对 auth 聚合无依赖

## 备选方案

| 方案 | 优点 | 缺点 | 排除原因 |
|------|------|------|----------|
| 单一 User 聚合（包含认证信息） | 查询简单 | 事务边界模糊、并发冲突高、职责不单一 | 违背聚合设计原则 |
| auth 作为 user 的子实体 | 天然关联 | 认证信息修改会锁定整个 user 聚合、扩展性差 | 事务粒度不合理 |
| 完全独立无关联 | 彻底解耦 | 无法表达业务关联、查询困难 | 业务语义丢失 |

## 影响

### 正向影响
- **职责清晰**：user 聚合专注用户资料，auth 聚合专注认证状态
- **事务边界合理**：登录失败锁定只需修改 AuthUser，不影响 User
- **独立演进**：两个聚合可独立优化、扩展、部署
- **避免循环依赖**：单向依赖（auth -> user 的 UserId）

### 负向影响
- 查询用户完整信息需要两次仓储查询
- 需要处理聚合间最终一致性（如用户删除时同步清理 auth 数据）

### 缓解措施
- Application 层编排用例时，显式处理两个聚合的协作
- 必要时引入领域事件实现跨聚合同步
