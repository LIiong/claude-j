# 质保文档

## 1. 测试策略

### 测试金字塔
```
        ┌───────────┐
        │  API/E2E  │  ← 少量、慢速、高置信度
        │   测试     │
        ├───────────┤
        │  集成测试   │  ← 适量，基础设施层
        │           │
        ├───────────┤
        │  单元测试   │  ← 大量、快速，domain + application
        │           │
        └───────────┘
```

### 各层测试标准

| 层 | 测试类型 | 框架 | 数据库 | Spring |
|---|---------|------|------|--------|
| Domain | 单元测试 | JUnit 5 + AssertJ | 无 | 无 |
| Application | 单元 + Mock | JUnit 5 + Mockito | 无 | 无 |
| Infrastructure | 集成测试 | @SpringBootTest | H2 | 是 |
| Adapter | API 测试 | @WebMvcTest + MockMvc | 无 | 部分 |

## 2. 覆盖率要求

| 层 | 最低覆盖率 | 重点领域 |
|---|----------|---------|
| Domain | 90%+ | 聚合不变量、状态转换、值对象 |
| Application | 80%+ | 用例编排、命令处理 |
| Infrastructure | 70%+ | Repository 往返、DO 转换 |
| Adapter | 70%+ | 请求校验、错误响应 |

## 3. 代码审查清单

### 架构合规性
- [ ] 模块依赖方向：`adapter -> application -> domain <- infrastructure`
- [ ] Domain 模块无 Spring/框架导入
- [ ] 业务逻辑在领域层，不在控制器或基础设施层
- [ ] Repository 接口（端口）在 domain，实现在 infrastructure
- [ ] 应用服务编排领域对象（薄层）

### DDD 模式合规性
- [ ] 聚合根封装不变量（非贫血模型）
- [ ] 值对象不可变，重写 equals/hashCode
- [ ] 状态变更仅通过聚合根方法
- [ ] 领域事件用于跨聚合通信
- [ ] 充血领域模型 — 逻辑在数据所在处

### 编码规范
- [ ] Java 8 兼容语法
- [ ] 各层 Lombok 注解使用正确
- [ ] 命名规范已遵循
- [ ] 对象转换链正确（DO ↔ Domain ↔ DTO ↔ Request/Response）
- [ ] 使用 MapStruct 进行转换
- [ ] 无未使用的 import 或死代码

### 安全性
- [ ] 无 SQL 注入风险（参数化查询）
- [ ] 所有 API 端点有输入校验（@Valid）
- [ ] 日志中无敏感数据
- [ ] 错误信息规范（无堆栈跟踪暴露给客户端）

## 4. 缺陷管理

### 严重级别
| 级别 | 描述 | SLA |
|------|------|-----|
| Critical | 架构违规、数据损坏、安全问题 | 验收前必须修复 |
| Major | 逻辑错误、缺失校验、行为不正确 | 验收前必须修复 |
| Minor | 风格问题、命名不一致、缺失注释 | 可后续修复 |

### 缺陷生命周期
```
发现 → 待修复 → 已修复 → 已验证 → 关闭
```

### 缺陷报告格式（在 test-report.md 中）
| # | 描述 | 严重级别 | 文件:行号 | 状态 |
|---|-----|---------|----------|------|
| 1 | 问题描述 | Critical/Major/Minor | path:line | 待修复/已修复/已验证 |

## 5. 回归测试

### 何时执行回归测试
- 任何缺陷修复后
- 代码重构后
- 验收任务完成前

### 回归范围
- 所有领域单元测试（始终执行）
- 受变更影响的应用层测试
- DO/Mapper 变更时的基础设施测试
- 端点行为变更时的 API 测试

## 6. QA 验收标准

任务被标记为"验收通过"需满足以下全部条件：
1. 所有单元测试通过（`mvn test`）
2. 无 Critical 或 Major 缺陷未关闭
3. 代码审查清单无阻塞项
4. 代码风格检查通过
5. 测试覆盖率达到最低阈值
6. 架构合规性验证通过（无依赖方向违反）
